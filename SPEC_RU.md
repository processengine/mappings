# Нормативная спецификация @processengine/mappings

**Версия:** 1.0.0  
**Статус:** нормативный документ  
**Язык реализации:** JavaScript (Node.js >= 18)

---

## 1. Назначение и границы

### 1.1 Что делает библиотека

`@processengine/mappings` это декларативная библиотека трансформации и нормализации JSON-данных. Она позволяет описать по конфигурации, как из одного или нескольких JSON-объектов собрать новый JSON и привести значения к каноническому виду.

Библиотека решает задачи:

- перенести поле из источника в результат;
- выбрать первое доступное значение из нескольких альтернатив;
- собрать новую структуру данных из разнородного входа;
- привести строковые значения к стандартному виду (регистр, пробелы, цифры);
- унифицировать несколько эквивалентных представлений одного значения через словарь;
- применить последовательность преобразований к одному значению.

### 1.2 Что библиотека не делает

Следующее намеренно исключено и не будет добавлено:

- условные операторы (`if`/`else`) общего назначения;
- арифметика;
- регулярные выражения;
- нормализация дат;
- нормализация телефонных номеров с автоопределением страны;
- контекстно-зависимые словари (выбор словаря по типу клиента, каналу и т.п.);
- предметная интерпретация данных — то есть логика, отвечающая на вопрос «что это значение означает для процесса», а не «как привести его к канонической форме»;
- вычисление бизнес-фактов из нескольких предметных условий;
- вложенные `transform`;
- `eval` и сетевые вызовы.

### 1.3 Архитектурное место

```text
@processengine/mappings    → собрать данные и нормализовать их перед следующим шагом
@processengine/rules       → применить правила к нормализованным данным
@processengine/decisions   → принять решение по сценарию
```

## 2. Основные понятия

### 2.1 Сценарий маппинга (definition)

Сценарий маппинга — это plain JSON-объект, описывающий:

- `mappingId` — уникальный идентификатор сценария (строка);
- `sources` — объявление источников данных: словарь `{ имя: "object" }`;
- `output` — словарь правил построения результата: каждый ключ — целевой путь, каждое значение — объект с ровно одним оператором.

### 2.2 Источники данных (sources)

При выполнении (`run`) передаётся словарь `{ имя: объект }`, где каждый объект соответствует объявленному источнику. Источники должны быть plain-объектами с JSON-safe содержимым.

### 2.3 Инвариант: один оператор на поле

Каждое поле в `output` содержит ровно один оператор. Комбинирование нескольких операторов в одном поле запрещено и отвергается валидатором.

### 2.4 Единая семантика создания поля

Каждый оператор производит ровно один из двух исходов:

- `outputCreated = true` — поле записывается в результат с конкретным значением;
- `outputCreated = false` — поле в результат не записывается.

Промежуточного состояния нет. Пустая строка `""` и `null` — это `outputCreated = true` с соответствующим значением. Только явный `outputCreated = false` означает отсутствие поля.

**Общие условия `outputCreated = false` для операторов, работающих с путями:**

- путь не разрешился (нет ключа в источнике, промежуточный узел не является plain object);
- значение по пути равно `null` — для операторов, требующих строку;
- значение по пути не является строкой — для строковых операторов и `mapValue`.

**Исключение из правила «null → не создавать»:** оператор `from` копирует `null` как есть — поле создаётся со значением `null`. Операторы `exists` и `equals` всегда создают поле (возвращают boolean).

## 3. Модель путей

### 3.1 Исходный путь (source path)

Формат: `sources.<sourceName>.<field>[.<nested>...]`

Правила:

- обязан начинаться с `sources.`;
- содержать минимум три сегмента: `sources`, имя источника, имя поля;
- не содержать пустых сегментов;
- не содержать запрещённых сегментов: `__proto__`, `prototype`, `constructor`;
- не содержать числовых сегментов (числовой индекс массива не поддерживается в v1).

Нарушение любого правила → ошибка `INVALID_PATH` при валидации.

### 3.2 Целевой путь (target path)

Формат: `<field>[.<nested>...]`

Правила совпадают с исходным путём, за исключением требования к префиксу `sources.`.

Нарушение → ошибка `INVALID_TARGET_PATH` при валидации.

При записи промежуточные объекты создаются автоматически. Конфликт путей (один является префиксом другого) → ошибка `CONFLICTING_TARGET_PATHS`.

### 3.3 Разрешение пути

Алгоритм:

1. Взять имя источника из второго сегмента. Если источника нет в `sources` → `resolved: false`.
2. Пройти по промежуточным сегментам. Если узел не является plain object или ключ отсутствует → `resolved: false`.
3. Проверить последний сегмент. Если ключ отсутствует → `resolved: false`.
4. Если последний ключ существует и его значение равно `null` → `resolved: true, value: null`.
5. Иначе → `resolved: true, value: <значение>`.

## 4. Операторы структурного маппинга

### 4.1 `from`

Копирует значение из источника по пути.

**Синтаксис:**

```json
"target.field": { "from": "sources.<n>.<field>" }
```

**Семантика:**

- путь не разрешился → `outputCreated = false`;
- путь разрешился → `outputCreated = true`, значение — deep copy исходного значения;
- `null` копируется как есть (`outputCreated = true, value: null`).

Deep copy гарантирует: изменение результата не мутирует источник. Используется рекурсивный обход с детектированием циклических ссылок. `JSON.stringify/parse` не используется.

### 4.2 `literal`

Вставляет константное значение в результат.

**Синтаксис:**

```json
"target.field": { "literal": <JSON-safe literal> }
```

JSON-safe literal: string, конечное число (не NaN, Infinity, -Infinity), boolean, null. Объекты и массивы не допускаются.

**Семантика:** всегда `outputCreated = true`.

### 4.3 `exists`

Проверяет, разрешается ли путь в не-null значение.

**Архитектурная граница:** `exists` предназначен только для вычисления локальных технических признаков на основе одного значения. Логические композиции (`and`, `or`, `not`) поверх результатов `exists` в этой библиотеке не поддерживаются и добавляться не будут. Составные бизнес-факты вычисляются в `@processengine/rules`.

**Синтаксис:**

```json
"target.field": { "exists": "sources.<n>.<field>" }
```

**Семантика:** всегда `outputCreated = true`, значение — boolean.

- `true` — путь разрешился и значение не равно `null`;
- `false` — путь не разрешился или значение равно `null`.

Замечание: значения `0`, `false`, `""` дают `true` — они не являются `null`.

### 4.4 `equals`

Строгое сравнение значения из источника с литералом (`===`).

**Архитектурная граница:** `equals` предназначен только для вычисления локальных технических признаков из одного значения. Логические композиции не поддерживаются и добавляться не будут.

**Синтаксис:**

```json
"target.field": { "equals": ["sources.<n>.<field>", <JSON-safe literal>] }
```

**Семантика:** всегда `outputCreated = true`, значение — boolean.

- `true` — путь разрешился и `value === literal`;
- `false` — путь не разрешился или `value !== literal`.

Нет приведения типов: `"42" !== 42`.

### 4.5 `coalesce`

Возвращает первое не-null значение из списка кандидатов.

**Синтаксис:**

```json
"target.field": {
  "coalesce": [
    { "path": "sources.<n>.<field>" },
    { "path": "sources.<m>.<field>" },
    { "literal": <JSON-safe literal> }
  ]
}
```

Допустимо от 1 до 4 кандидатов. Каждый кандидат — plain object с ровно одним ключом: `path` или `literal`.

**Семантика:**

Кандидаты проверяются слева направо.

- Path-кандидат: если путь разрешился и значение не равно `null` → выбирается это значение (deep copy).
- Literal-кандидат: всегда совпадает. Не обходится дальше.

Если ни один кандидат не выбран → `outputCreated = false`.

## 5. Операторы форматной нормализации

Все строковые операторы являются корневыми операторами поля. Принимают путь к источнику. Работают только со строковыми значениями.

**Общий контракт:**

- путь не разрешился → `outputCreated = false`;
- значение null → `outputCreated = false`;
- значение не строка (число, boolean, объект, массив) → `outputCreated = false`;
- значение строка → применить преобразование, `outputCreated = true`.

Нет неявного приведения типов. Число `42` не приводится к строке `"42"` перед обработкой.

### 5.1 `trim`

Убирает ведущие и хвостовые пробельные символы: пробел (`\u0020`), `\t`, `\n`, `\r`.

```json
"client.name": { "trim": "sources.req.rawName" }
```

Пример: `"  Иван  "` → `"Иван"`.

### 5.2 `lowercase`

Приводит к нижнему регистру через `String.prototype.toLowerCase()`.

```json
"client.email": { "lowercase": "sources.req.email" }
```

Пример: `"CUSTOMER@EXAMPLE.COM"` → `"customer@example.com"`.

### 5.3 `uppercase`

Приводит к верхнему регистру через `String.prototype.toUpperCase()`.

```json
"doc.type": { "uppercase": "sources.doc.rawType" }
```

Пример: `"passport"` → `"PASSPORT"`.

### 5.4 `normalizeSpaces`

Trim + замена всех внутренних последовательностей пробельных символов на один пробел (`\u0020`).

```json
"client.fullName": { "normalizeSpaces": "sources.req.name" }
```

Пример: `"  Иван   Иванов  "` → `"Иван Иванов"`.

### 5.5 `removeNonDigits`

Удаляет все символы, кроме `[0-9]`.

```json
"client.phone": { "removeNonDigits": "sources.req.rawPhone" }
```

Пример: `"+7 (999) 111-22-33"` → `"79991112233"`.

**Особый случай:** если после фильтрации строка стала пустой — создаётся поле со значением `""`. Это `outputCreated = true`. Пустая строка не эквивалентна отсутствию поля.

Пример: `"---"` → `""`.

## 6. Оператор `mapValue`

Оператор `mapValue` ищет строковое значение в явно заданном словаре и возвращает соответствующее канонизированное значение.

**Ключевой принцип:** `mapValue` не выполняет неявного приведения типов. Поиск выполняется только если входное значение является строкой. Число `643` и строка `"643"` — разные значения; оператор не будет искать `"643"` при числовом входе.

### 6.1 Корневая форма

Используется как самостоятельный оператор поля.

**Синтаксис:**

```json
"payment.currency": {
  "mapValue": {
    "from":     "sources.req.currencyCode",
    "map":      { "RUR": "RUB", "643": "RUB", "840": "USD", "978": "EUR" },
    "fallback": "passthrough"
  }
}
```

**Поля:**

`from` — обязательное, строка, путь к исходному значению. Те же правила валидации, что у `from`.

`map` — обязательное, plain object. Ключи — строки. Значения — JSON-safe literals. Не может быть пустым.

`fallback` — опциональное. Определяет поведение **только в случае `no_match`** — когда путь разрешился в строку, но эта строка не найдена как ключ в словаре. При `path_unresolved`, `null_value` или `type_mismatch` поле не создаётся, `fallback` не применяется (см. 6.3).

**Семантика:**

1. Разрешить `from`. Если путь не разрешился, значение `null` или не строка → `outputCreated = false` (fallback **не применяется**).
2. Искать строковое значение в `map`. Поиск строгий: `"643"` ≠ `643`.
3. Ключ найден → `outputCreated = true`, значение из словаря.
4. Ключ не найден → применить `fallback`.

### 6.2 Шаговая форма

Используется только как шаг внутри `transform.steps`. Отличается от корневой формы отсутствием поля `from` — входное значение берётся из результата предыдущего шага.

**Синтаксис внутри steps:**

```json
{ "mapValue": { "map": { "М": "MALE", "M": "MALE" }, "fallback": null } }
```

Поле `from` в шаговой форме отсутствует и не допускается.

### 6.3 Семантика `fallback`

| Значение `fallback` | Поведение при ненайденном ключе                    |
| ------------------- | -------------------------------------------------- |
| отсутствует         | `outputCreated = false`, поле не создаётся         |
| `null`              | `outputCreated = true`, значение `null`            |
| JSON-safe literal   | `outputCreated = true`, значение — этот литерал    |
| `"passthrough"`     | `outputCreated = true`, значение — исходная строка |

В шаговой форме при `outputCreated = false` цепочка прерывается.

## 7. Оператор `transform`

`transform` — самостоятельный корневой оператор поля. Применяет последовательность из 2–8 шагов к значению, взятому из источника.

**`transform` не комбинируется с другими корневыми операторами.** Правило «ровно один оператор на поле» сохраняется.

**Когда использовать:** когда нужно два и более последовательных преобразования. Если достаточно одного — используйте соответствующий корневой оператор.

### 7.1 Синтаксис

```json
"client.gender": {
  "transform": {
    "from":  "sources.raw.gender",
    "steps": [
      { "trim":      true },
      { "uppercase": true },
      { "mapValue": {
          "map":      { "М": "MALE", "M": "MALE", "Ж": "FEMALE", "F": "FEMALE" },
          "fallback": null
      }}
    ]
  }
}
```

### 7.2 Поля

`from` — обязательное, строка, путь к исходному значению. Семантика совпадает с оператором `from`.

`steps` — обязательное, массив, **2–8 шагов**. Нижняя граница 2: для одного шага используйте соответствующий корневой оператор.

### 7.3 Допустимые шаги

**Флажковые шаги** — аргумент `true`:

```json
{ "trim":             true }
{ "lowercase":        true }
{ "uppercase":        true }
{ "normalizeSpaces":  true }
{ "removeNonDigits":  true }
```

**Параметрический шаг** — `mapValue` в шаговой форме:

```json
{ "mapValue": { "map": { ... }, "fallback": ... } }
```

Операторы `from`, `literal`, `exists`, `equals`, `coalesce`, `transform` внутри `steps` не допускаются. Вложенные `transform` не допускаются.

### 7.4 Семантика выполнения

1. Разрешить `from`. Если путь не разрешился или значение `null` → `outputCreated = false`, цепочка не запускается.
2. Выполнять шаги последовательно. Входное значение первого шага — результат разрешения `from`. Входное значение каждого последующего шага — результат предыдущего.
3. Если шаг не может создать значение (`outputCreated = false`) → цепочка прерывается, `outputCreated = false` на уровне `transform`.
4. Результат последнего шага записывается в output.

**Условия прерывания цепочки:**

- не-строковое значение на входе строкового шага → `reason: "type_mismatch"`;
- `mapValue`-шаг: ключ не найден и `fallback` отсутствует → `reason: "no_match"`.

### 7.5 Важно о null внутри цепочки

Если `mapValue`-шаг возвращает `null` (через `fallback: null`) и это не последний шаг — следующий строковый шаг получит `null` как вход, что приведёт к `type_mismatch` и обрыву цепочки. Это корректное поведение.

Если `mapValue`-шаг с `fallback: null` является последним шагом — `transform` создаёт поле со значением `null`.

## 8. Известные ограничения v1

### 8.1 Массивы не поддерживаются

Пути с числовыми индексами (`sources.a.items.0`) отвергаются при валидации. Обход, нормализация или выборка элементов массива по индексу недоступны.

Это сознательная граница версии, а не забытый случай. Библиотека v1 рассчитана на скалярные и объектные поля.

Планируется в v2: ограниченная поддержка нормализации коллекций без общего языка итераций и без условий общего назначения.

### 8.2 Даты и телефоны

Нормализация дат и телефонных номеров (с автоопределением страны) выходит за рамки v1. `removeNonDigits` позволяет извлечь цифры из телефонного номера, но не выполняет предметной интерпретации.

## 9. Трассировка

Трассировка включается флагом `trace: true` в `run()`. `validate()` никогда не возвращает `trace`.

```js
const result = engine.run({ definition, sources, trace: true });
// result.trace — массив записей, по одной на каждое поле output
```

### 9.1 Структура записи (базовая)

```json
{
  "target": "client.name",
  "op": "trim",
  "path": "sources.req.rawName",
  "resolved": true,
  "inputValue": "  Иван  ",
  "outputCreated": true,
  "outputValue": "Иван"
}
```

При `outputCreated = false` поле `outputValue` отсутствует; присутствует поле `reason`.

### 9.2 Запись для `mapValue`

```json
{
  "target": "payment.currency",
  "op": "mapValue",
  "path": "sources.req.currencyCode",
  "resolved": true,
  "inputValue": "RUR",
  "matched": true,
  "outputCreated": true,
  "outputValue": "RUB"
}
```

При `matched: false` добавляются поля `fallbackApplied` (boolean) и, если fallback применён, `fallbackKind`: `"literal"` или `"passthrough"`.

### 9.3 Запись для `transform` — happy path

```json
{
  "target": "client.gender",
  "op": "transform",
  "path": "sources.raw.gender",
  "resolved": true,
  "inputValue": " m ",
  "steps": [
    { "op": "trim", "in": " m ", "out": "m", "applied": true },
    { "op": "uppercase", "in": "m", "out": "M", "applied": true },
    {
      "op": "mapValue",
      "in": "M",
      "out": "MALE",
      "applied": true,
      "matched": true
    }
  ],
  "outputCreated": true,
  "outputValue": "MALE"
}
```

### 9.4 Запись для `transform` — обрыв цепочки

Шаги до обрыва: `applied: true`. Шаг обрыва: `applied: false`, `stoppedChain: true`, `reason`. Шаги после: не включаются.

```json
{
  "target": "client.gender",
  "op": "transform",
  "path": "sources.raw.gender",
  "resolved": true,
  "inputValue": " UNKNOWN ",
  "steps": [
    { "op": "trim", "in": " UNKNOWN ", "out": "UNKNOWN", "applied": true },
    {
      "op": "mapValue",
      "in": "UNKNOWN",
      "applied": false,
      "stoppedChain": true,
      "reason": "no_match",
      "matched": false
    }
  ],
  "outputCreated": false,
  "reason": "chain_stopped"
}
```

При `from` не разрешился — поле `steps` отсутствует.

## 10. Модель ошибок

Оба метода (`validate`, `run`) всегда возвращают `MappingResult` и никогда не бросают исключения в штатных сценариях.

```js
// Успех
{ status: 'SUCCESS', mappingId: '...', result: { ... } }

// Ошибка
{ status: 'MAPPING_ERROR', mappingId: '...', error: { code: '...', message: '...' } }
```

Поле `mappingId` присутствует в ответе, если его удалось извлечь из `definition`.

### 10.1 Коды ошибок конфигурации (до исполнения)

| Код                          | Условие                                                                                                             |
| ---------------------------- | ------------------------------------------------------------------------------------------------------------------- |
| `INVALID_MAPPING_SCHEMA`     | `definition` не является plain object; `output` не plain object; поле output не имеет оператора или имеет несколько |
| `INVALID_MAPPING_ID`         | `mappingId` отсутствует, не строка или пустая строка                                                                |
| `UNKNOWN_OPERATOR`           | Неизвестный ключ оператора в поле output                                                                            |
| `INVALID_ARGS`               | Некорректные аргументы оператора                                                                                    |
| `INVALID_SOURCE_DECLARATION` | `sources` не plain object; тип источника не `"object"`                                                              |
| `INVALID_PATH`               | Нарушение синтаксиса исходного пути                                                                                 |
| `INVALID_TARGET_PATH`        | Нарушение синтаксиса целевого пути                                                                                  |
| `CONFLICTING_TARGET_PATHS`   | Один целевой путь является префиксом другого                                                                        |

### 10.2 Коды ошибок выполнения (во время исполнения)

| Код                      | Условие                                                                                                     |
| ------------------------ | ----------------------------------------------------------------------------------------------------------- |
| `MISSING_SOURCE`         | Объявленный источник не передан в `sources`                                                                 |
| `INVALID_SOURCE_TYPE`    | `sources` не plain object; переданный источник не plain object                                              |
| `INVALID_SOURCE_CONTENT` | Источник содержит не-JSON-safe значения: NaN, Infinity, Date, Map, Set, BigInt, функции, циклические ссылки |
| `INTERNAL_ERROR`         | Непредвиденная ошибка внутри библиотеки                                                                     |

## 11. Безопасность

### 11.1 Prototype pollution

Сегменты `__proto__`, `prototype`, `constructor` запрещены как в исходных, так и в целевых путях. Проверяется при валидации.

При записи результата используется `Object.prototype.hasOwnProperty.call()` для проверки ключей — не `in`-оператор, который обходит прототипную цепочку.

### 11.2 JSON-safe контракт

Все источники данных проверяются рекурсивно перед исполнением. Запрещены:

- `NaN`, `Infinity`, `-Infinity`;
- `BigInt`;
- функции;
- `undefined`;
- экземпляры классов: `Date`, `Map`, `Set` и любые другие не-plain объекты;
- циклические ссылки.

### 11.3 Deep copy

`deepCopy` использует рекурсивный обход. Никогда не полагается на `JSON.stringify/parse`:

- `JSON.stringify` преобразует `NaN → null`, `Date → string` молча;
- `deepCopy` бросает `TypeError` при обнаружении не-JSON значений, что превращается в `INTERNAL_ERROR`.

### 11.4 Числовые индексы

Пути вида `sources.a.items.0` отвергаются при валидации. Доступ к элементам массива по индексу не поддерживается в v1.

## 12. Публичный API

### 12.1 `MappingEngine` и `compile`

```js
const {
  MappingEngine,
  compile,
  CompiledMapping,
} = require("@processengine/mappings");
```

#### Иммутабельность `CompiledMapping`

`compile(definition)` создаёт `CompiledMapping`, полностью независимый от исходного объекта `definition`. Внутреннее представление — глубокая копия с глубокой заморозкой (`JSON.parse(JSON.stringify(...))` + рекурсивный `Object.freeze`).

Гарантии:

- мутация объекта `definition` после `compile()` не влияет на поведение `execute()`;
- скомпилированный артефакт безопасен для многократного использования и переиспользования между вызовами;
- `execute()` всегда работает по состоянию definition на момент компиляции.

Этот контракт проверяется тестами в `test/contract.test.js`.

### 12.2 `engine.validate(definition)`

Проверяет описание маппинга без его выполнения.

Возвращает `MappingResult`:

- при успехе: `{ status: 'SUCCESS', mappingId }`
- при ошибке: `{ status: 'MAPPING_ERROR', mappingId?, error: { code, message } }`

Никогда не возвращает `result` или `trace`.

### 12.3 `engine.run({ definition, sources, trace? })`

Проверяет и выполняет маппинг.

Параметры:

- `definition` — объект описания маппинга;
- `sources` — словарь `{ имя: plain object }`;
- `trace` — boolean, по умолчанию `false`.

Возвращает `MappingResult`:

- при успехе: `{ status: 'SUCCESS', mappingId, result: { ... } }`
- при успехе с трассировкой: добавляется поле `trace: [...]`
- при ошибке: `{ status: 'MAPPING_ERROR', mappingId?, error: { code, message } }`

Никогда не бросает исключения.

### 12.4 Экспорт

Библиотека экспортирует только `{ MappingEngine }` через `src/index.js`. Поле `exports` в `package.json` ограничивает публичный API этим файлом.

---

#### Покрытие JSON Schema и runtime-validator

Формальная JSON Schema (`schema/mapping-definition.v1.schema.json`) является частью публичного контракта библиотеки. Она покрывает структурную форму всех операторов и типы аргументов.

Она **намеренно не покрывает** следующие инварианты — они требуют контекста выполнения и проверяются только runtime-validator (`compile()` / `validate()`):

- синтаксис исходных путей (`sources.<n>.<field>` с минимум тремя сегментами)
- перекрёстные ссылки: объявлен ли источник, на который ссылается путь
- конфликтующие целевые пути (один является префиксом другого)
- запрещённые сегменты в путях (`__proto__`, `prototype`, `constructor`, числовые индексы)

JSON Schema — правильный инструмент для структурной валидации и IDE-автодополнения. Для полной валидации сценария используйте `compile()` или `validate()`.

## 13. CLI

```bash
mappings validate-file <file>
mappings run-file <file> --sources <sources.json> [--trace]
mappings validate-dir <directory>
mappings list <directory>
```

`validate-dir` пропускает JSON-файлы без поля `mappingId` с пометкой в выводе. Возвращает код выхода `0` при успехе, `1` при наличии ошибок.

## 14. Обратная совместимость

Все операторы v1 (`from`, `literal`, `exists`, `equals`, `coalesce`) сохраняют прежнюю семантику без изменений. Инвариант «ровно один корневой оператор на поле» сохраняется. Новые операторы добавлены как дополнительные допустимые корневые операторы и не меняют поведения существующих сценариев.
